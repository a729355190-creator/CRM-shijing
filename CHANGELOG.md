# 仕净V6 更新日志

> 记录方式：按日期分组，每条对应一个功能点/修复点。commit哈希可用 `git show <hash>` 查看具体代码改动。

## 2026-07-18
- `4e6af09` **修复本地推账户列表刷新为空覆盖配置导致数据丢失的bug**：用户反馈"本地推消耗数据拉取异常"，排查确认7/15、7/17两天数据完全缺失。根因是`syncLocalAccountList()`每天同步前刷新账户列表，若巨量接口偶发返回空列表(非报错)，会直接用空数组覆盖现有配置并落库，导致账户数瞬间清零，当天同步循环因"无账户"空转但仍标记`ok:true`，日志上看不出任何异常。已修复：接口返回空且现有缓存非空时不再覆盖，明确返回失败信号；账户数为0或写入0条时推HQ企微群告警。已补齐7/15、7/17两天历史数据(各32条)，并用故障模拟回归测试验证修复生效。同时确认：本地推/ADQ的城市维度数据是从未实现过的新功能(账户/日报表接口本身不支持城市聚合，需要接入巨量受众分析接口和腾讯定向标签报表接口)，非近期故障，已建任务排期后续接入。

## 2026-07-15
- `0f6537d` **隐私加固：投放归因字段收紧为仅总部可见**：用户指出"客服/门店看到顾客来源平台/账户，可能导致跨渠道调顾客"。排查发现客户档案页(`/api/hub/customer/:ext`)三角色共用同一渲染逻辑，`source_city`(来源城市，2026-07-01上线时就存在)和刚新增的`attribution_channel`/`attribution_state`都对客服/门店可见。现收紧为只有`hq`角色才返回真实值，cs/store视角一律为null。已用真实数据验证：客户归属的客服本人也看不到来源信息。
- `8b5aae3` **客户中心-阶段4：打通投手归因地基**：给已有客户中心体系(客户主档`shijing_wecom_customers`)新增`attribution_channel`/`attribution_state`归因字段，新建`shijing_invite_attribution`归因快照表。客服排客时若选中企微好友，系统自动反查该好友身上的投放归因并存档，全程不需要手动选投手。客户档案页新增"投放归因"展示行。归因数据源头(获客链接customer_channel)尚未接入，字段暂为空。
- `1941d09` **账户级归因方案：ADQ账户接入认领机制 + 平台投产对比看板**：比"精确到每次点击"更易落地的降级方案——巨量AD/本地推早有的"账户认领"机制(2026-07-05)扩展支持腾讯ADQ账户，投放人员可在"账户认领"页直接勾选ADQ账户认领归属；新增"平台投产对比"看板，巨量AD/巨量本地推/腾讯ADQ三平台横向对比消耗/加粉/高潜/营业额(估算)/ROI(估算)。不依赖任何外部配置，账户级归因即时生效。
- `e80f974` **修复三处数据看板真实bug**：①总部数据看板"城市维度(消耗/营业额/ROI)"表格的"到店数"一直显示0、"到店成本"显示"-"——根因是接口聚合城市对象时漏初始化`arrivedCount`字段，门店新客到店数累加时变成`undefined++`产生NaN，前端兜底成了0（非数据源问题，用户建议的"引用门店新到数"口径本来就对，只是被这个初始化漏洞挡住了）；②"客服团队明细"页面"到店率"公式改为 已到店/加粉数（原先误写成 已到店/排客数），与其他页面口径统一；③"全链路看板"营业额此前用的是客户全生命周期累计总额（跟选7天还是30天无关，固定返回同一个数），改为只统计当前时间窗口内的到店事件营业额，客单价/ROI等衍生指标随之修正。
- `19e3ffa` **接入腾讯广告(ADQ)真实数据同步**：新增独立的OAuth授权+token管理模块，渠道分析页"腾讯ADQ"标签从"待授权"占位改为真实数据展示。已实测账户(长沙净须美容有限公司)30天历史消耗/曝光/点击/转化数据回填成功。
- `b582061` **修复本地推"高潜成交(预付定金)"一直显示0的问题**：此前代码把该字段硬编码为0（旧注释误判"巨量原生接口拿不到"）。实测证实巨量线索明细接口返回的每条线索都带`component_event_type_tags`标签数组，其中`196`即代表命中"预付定金"深度转化节点，数据其实一直都在，只是代码没解析。现改为按天统计含196标签的线索条数写入高潜成交，历史数据已补齐验证。
